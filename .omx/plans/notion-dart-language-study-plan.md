## 6. 권장 목차

### 6.1 Dart 언어 개요

설명할 내용:

- Dart는 Flutter의 기반 언어다.
- 정적 타입 언어이지만 `var`를 통해 타입 추론을 적극적으로 사용한다.
- null safety가 언어 차원에서 기본 제공된다.
- 객체지향 문법과 함수형 스타일 컬렉션 조작을 함께 사용한다.

C#/Java 비교:

- C#과 Java처럼 클래스 기반 객체지향 언어에 가깝다.
- JavaScript처럼 런타임에 타입이 자주 바뀌는 언어와는 다르게, 기본적으로 타입 안정성을 추구한다.
- C#의 nullable reference type보다 Dart의 null safety가 문법 전반에 더 강하게 드러난다.

Flutter 연결:

- Flutter 위젯은 Dart 클래스와 생성자 문법을 강하게 사용한다.
- named parameter, `const`, null safety, 컬렉션 spread는 Flutter 코드에서 매우 자주 등장한다.

### 6.2 Dart 프로그램의 시작점

기준 파일:

- `ex01/hello_world.dart`

정리할 내용:

- Dart 프로그램은 `main()` 함수에서 시작한다.
- `void main()`은 반환값이 없는 프로그램 진입점이다.
- `print()`는 콘솔 출력에 사용한다.

예시 코드:

```dart
void main() {
  print("Hello World!");
}
```

주의할 점:

- Flutter 앱에서도 내부적으로는 `main()`에서 앱 실행이 시작된다.
- 나중에 Flutter에서는 `runApp()`을 `main()` 안에서 호출하게 된다.

### 6.3 변수와 기본 타입

기준 파일:

- `ex01/test01.dart`
- `ex01/test02.dart`

정리할 내용:

- Dart의 대표 기본 타입: `int`, `double`, `bool`, `String`
- `runtimeType`으로 런타임 타입을 확인할 수 있다.
- `var`는 타입을 추론하지만, 한 번 정해진 타입은 바뀌지 않는다.
- `dynamic`은 다른 타입으로 바꿀 수 있지만 정적 타입 검사의 장점을 잃는다.
- `final`은 한 번만 값을 할당할 수 있는 변수다.

비교 표:

| 문법 | 타입 고정 여부 | 값 재할당 | 다른 타입 재할당 | 권장 사용 |
| --- | --- | --- | --- | --- |
| 명시 타입 | 고정 | 가능 | 불가능 | 공개 API, 의도를 분명히 할 때 |
| `var` | 고정 | 가능 | 불가능 | 지역 변수 |
| `dynamic` | 고정되지 않음 | 가능 | 가능 | 외부 데이터처럼 타입을 확정하기 어려울 때만 |
| `final` | 고정 | 불가능 | 불가능 | 한 번 정한 값을 바꾸면 안 될 때 |

C#/Java 비교:

- `var`는 C#의 `var`와 비슷하게 컴파일 시점에 타입이 정해진다.
- `dynamic`은 C#의 `dynamic`과 유사하게 타입 검사를 런타임으로 미루지만, 남용하면 안전성이 떨어진다.
- `final`은 Java의 `final` 지역 변수와 비슷하게 재할당을 막는다.

Flutter 연결:

- Flutter에서는 `final` 필드를 가진 위젯 클래스를 자주 작성한다.
- 상태를 직접 바꾸기보다 새 객체를 만들어 교체하는 패턴이 많기 때문에 `final`과 불변성 이해가 중요하다.

주의할 점:

- `dynamic`은 편해 보이지만 자동완성, 정적 분석, 타입 안정성을 약하게 만든다.
- Dart 학습 초반에는 `dynamic`보다 `var` 또는 명시 타입을 우선 사용한다.

### 6.7 Cascade operator

기준 파일:

- `ex01/test07.dart`

정리할 내용:

- `..`는 같은 객체에 연속해서 접근할 때 사용하는 cascade operator다.
- 메서드를 호출한 뒤에도 원래 객체를 반환하는 흐름을 만들 수 있다.

예시:

```dart

```

C#/Java 비교:

- C#과 Java에는 Dart의 cascade operator와 같은 문법이 기본적으로 없다.
- 빌더 패턴의 체이닝과 비슷하게 보일 수 있지만, `..`는 메서드가 자기 자신을 반환하지 않아도 같은 객체에 이어서 접근하게 해준다.

Flutter 연결:

- Flutter 코드에서는 자주 보이는 핵심 문법은 아니지만, 객체 초기화나 테스트 코드에서 가독성을 높일 수 있다.

### 6.8 Null safety

기준 파일:

- `ex01/test08.dart`
- `ex01/test09.dart`
- `ex01/test10.dart`
- `ex01/test11.dart`

정리할 내용:

- Dart는 기본적으로 null이 들어갈 수 없는 타입을 사용한다.
- null을 허용하려면 타입 뒤에 `?`를 붙인다.
- `?.`는 null이면 접근하지 않고 null을 반환한다.
- `??`는 왼쪽 값이 null이면 오른쪽 기본값을 사용한다.
- `!`는 개발자가 “이 값은 절대 null이 아니다”라고 강제로 단언하는 문법이다.
- `late`는 선언 시점에는 값을 넣지 않지만, 사용 전에는 반드시 초기화하겠다고 약속하는 문법이다.

C#/Java 비교:

- Java는 기본적으로 참조 타입에 null이 들어갈 수 있고, 컴파일러가 강하게 막아주지 않는다.
- C#의 nullable reference type과 개념적으로 비슷하지만 Dart는 `?`, `!`, `late` 문법이 더 일상적으로 사용된다.

Flutter 연결:

- Flutter에서 위젯 매개변수, 상태 값, 비동기 데이터는 null 가능성을 자주 다룬다.
- null safety를 이해하지 못하면 Flutter 코드에서 `?`, `!`, `required`, `late`를 볼 때 계속 막힌다.

주의할 점:

- `!`는 오류를 없애는 마법 문법이 아니다. null이면 런타임 오류가 난다.
- `late`도 초기화하지 않고 사용하면 런타임 오류가 난다.
- 가능하면 `??`로 기본값을 제공하거나, null 여부를 검사한 뒤 사용하는 편이 안전하다.

현재 예제에서 짚을 점:

- `test10.dart`의 `String? name = "Jhon"; name!`은 실제로 null이 될 수 없는 값이라 학습용으로는 약하다. null 가능성이 있는 함수 반환값에 `!`를 쓰는 예제로 바꾸면 개념이 더 분명해진다.

기준 파일:

- `ex01/test12.dart`
- `ex01/test13.dart`
- `ex01/test14.dart`
- `ex01/test15.dart`

정리할 내용:

- 자식 객체는 자식 타입이면서 동시에 부모 타입으로도 볼 수 있다.
- 부모 생성자는 자식 생성자 본문보다 먼저 실행된다.
- `final` 필드는 생성자 실행 과정에서 반드시 초기화되어야 한다.
- 부모 생성자에 값을 넘길 때는 `: super(name)` 형태를 사용한다.

C#/Java 비교:

- `extends`와 `super`의 기본 개념은 Java와 비슷하다.
- C#의 base constructor 호출과도 비슷하다.
- Dart에서는 생성자 초기화 리스트 문법이 중요하다.

### 6.10 추상 클래스, implements, 동적 바인딩

기준 파일:

- `ex01/test17.dart`

정리할 내용:

- `abstract class`는 직접 인스턴스화하기보다 공통 규약을 정의할 때 사용한다.
- `implements`는 해당 클래스가 특정 규약을 구현하겠다는 의미다.
- `Animal` 타입으로 `Dog`, `Cat`, `Fish`를 받을 수 있고, 실행 시점에는 실제 객체의 메서드가 실행된다.
- 이것이 동적 바인딩이다.

C#/Java 비교:

- Java의 interface 구현과 비슷하게 이해할 수 있다.
- Dart에서는 모든 class가 암묵적으로 interface 역할도 할 수 있다는 점이 중요하다. 이 내용은 이후 학습에서 더 확장한다.

### 6.11 Mixin

기준 파일:

- `ex01/test16.dart`

정리할 내용:

- mixin은 상속 계층을 만들지 않고 기능을 섞어 넣는 문법이다.
- `with Engine, Wheel`처럼 여러 기능을 조합할 수 있다.
- `mixin class`는 mixin으로도 쓰고 클래스처럼 인스턴스화할 수도 있다.

C#/Java 비교:

- Java에는 Dart mixin과 같은 문법이 없다.
- C#의 interface default implementation이나 extension method와 일부 목적이 비슷해 보일 수 있지만, Dart mixin은 클래스에 상태와 메서드를 실제로 섞어 넣는 방식에 가깝다.

Flutter 연결:

- Flutter에서는 `SingleTickerProviderStateMixin`처럼 mixin을 자주 만난다.
- 애니메이션, 상태 보조 기능 등을 조합할 때 mixin 개념이 중요하다.

### 6.12 컬렉션

기준 파일:

- `ex01/test18.dart`
- `ex01/test19.dart`

정리할 내용:

- `List`: 순서가 있고 인덱스로 접근한다.
- `Map`: key-value 구조다.
- `Set`: 중복을 허용하지 않는다.
- `for` 반복문으로 직접 순회할 수 있다.
- `map`은 각 요소를 변환한다.
- `where`는 조건에 맞는 요소만 걸러낸다.
- `Set`을 `List`로 바꾼 뒤 정렬할 수 있다.

C#/Java 비교:

- `List`는 C#의 `List<T>`, Java의 `List<T>`와 비슷하다.
- `Map`은 C#의 `Dictionary<TKey, TValue>`, Java의 `Map<K, V>`와 비슷하다.
- `map`, `where`는 C# LINQ의 `Select`, `Where`와 비슷한 느낌으로 이해할 수 있다.

Flutter 연결:

- Flutter에서 리스트 UI를 만들 때 컬렉션 변환이 자주 쓰인다.
- 예를 들어 데이터 리스트를 위젯 리스트로 바꿀 때 `map`을 사용한다.

현재 예제에서 짚을 점:

- 로또 예제는 `Set`을 사용하므로 중복 숫자가 나오면 6개보다 적게 담길 수 있다.
- 정확히 6개의 고유 숫자를 원한다면 `while (lotto.length < 6)` 같은 반복 구조가 필요하다.

### 6.13 Spread operator와 복사

기준 파일:

- `ex01/test20.dart`

정리할 내용:

- `...list`는 기존 컬렉션의 요소를 새 컬렉션 안에 펼쳐 넣는다.
- 단순 리스트는 spread로 쉽게 복사할 수 있다.
- 하지만 리스트 안에 map 같은 참조 객체가 들어 있으면 얕은 복사 문제가 생긴다.
- 내부 map까지 새로 만들려면 `list2.map((i) => {...i}).toList()`처럼 각 요소를 복사해야 한다.
- `{...user, "username": "love"}`는 기존 map의 값을 유지하면서 일부 값만 바꾸는 패턴이다.

C#/Java 비교:

- C#의 spread와 유사한 컬렉션 표현식 또는 LINQ 변환과 비교할 수 있다.
- Java에서는 stream과 새 객체 생성을 조합해야 비슷한 패턴을 만들 수 있다.

Flutter 연결:

- Flutter 상태 관리에서는 기존 상태를 직접 바꾸기보다 새 리스트나 새 map을 만들어 교체하는 방식이 중요하다.
- 이 예제는 나중에 state update 패턴으로 이어진다.

### 6.14 `const` 생성자와 객체 재사용

기준 파일:

- `ex01/test21.dart`

정리할 내용:

- `const` 생성자는 컴파일 타임 상수 객체를 만들 수 있게 한다.
- 같은 `const` 생성자와 같은 인자를 사용하면 Dart는 동일한 객체를 재사용할 수 있다.
- `final`은 재할당을 막는 것이고, `const`는 컴파일 타임 상수를 만든다는 점에서 다르다.

C#/Java 비교:

- Java의 `final`과 Dart의 `const`는 같은 개념이 아니다.
- C#의 `const`와도 일부 유사하지만, Dart에서는 객체 생성자에도 `const`를 붙일 수 있다는 점이 중요하다.

Flutter 연결:

- Flutter에서 `const Widget`을 잘 사용하면 불필요한 rebuild 비용을 줄이는 데 도움이 된다.
- 따라서 Dart의 `const` 개념은 Flutter 성능과도 연결된다.

## 7. 원본 예제 보존 원칙

현재 Dart 예제들은 책을 보며 작성한 학습 코드이므로, 노션 정리에서는 예제를 함부로 고치거나 비판하는 방향으로 쓰지 않는다. 각 예제는 해당 시점에 어떤 문법을 익히기 위한 코드인지 중심으로 설명한다.

코드 수정은 아래 기준에 맞을 때만 한다.

- 클래스명, 생성자명, 변수명처럼 명백한 오타가 있어 학습 흐름을 방해하는 경우
- 오타 수정으로 예제의 의도나 구조가 바뀌지 않는 경우
- 문법 설명을 위해 꼭 필요한 경우

그 외의 스타일 차이, 더 나은 예제 작성법, analyzer warning은 별도 개선 항목으로 정리하지 않는다. 나중에 리팩터링이나 예제 정리를 따로 하기로 결정한 경우에만 별도 문서에서 다룬다.
